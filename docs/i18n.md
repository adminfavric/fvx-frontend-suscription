# Internacionalización (i18n) — `fvx-frontend` + API

**Vista Django (gettext, `role_label`, `locale/`):** [`../../fvx-backend/docs/i18n.md`](../../fvx-backend/docs/i18n.md).

La UI usa **[@jsverse/transloco](https://github.com/jsverse/transloco)** con JSON por idioma bajo `public/assets/i18n/{lang}.json`. El idioma activo se persiste en `localStorage` con la clave **`fvx_lang`** (`en` | `es`). El valor por defecto de la app es **`en`**.

## Convención de claves

- Claves anidadas con punto: `layout.profile`, `crud.addItem`, `dataTable.refresh`.
- Interpolación en JSON: `{{param}}` (ej. `crud.addItem`: `"Add {{name}}"` / `"Añadir {{name}}"` con `transloco` pipe y `{ name: entityName }` en `app-crud-page`).
- **Fallback:** `fallbackLang` y `defaultLang` están en `en` en `app.config.ts`; si falta una clave en `es`, se usa la cadena de `en` cuando exista.
- Textos de ayuda en formularios CRUD (p. ej. `users.form.staffRestrictedHint`) siguen la misma convención bajo la clave del feature (`users.*`).

## Header `Accept-Language`

El interceptor `src/app/core/interceptors/locale.interceptor.ts` añade **`Accept-Language: <idioma activo>`** solo a peticiones cuya URL empieza por `environment.apiUrl` o `environment.authUrl`, para que Django resuelva `gettext` y etiquetas como `role_label` en los serializers.

Orden de interceptores en `app.config.ts`: **locale → auth → httpLogging**.

## Contrato API: `role_label`

El backend expone códigos de rol en inglés (`ADMIN`, `EDITOR`, `VIEWER`) y, en representaciones de perfil (`User.profile`), **`role_label`** (texto traducido según `Accept-Language`) junto con **`role`** efectivo.

Las etiquetas vienen de `gettext_lazy` en `fvx-backend/api/choices.py` (`ROLE_CHOICES`). Tras cambiar cadenas traducibles en el backend, **regenera y compila gettext con Docker** desde el directorio del backend (servicio `web`):

```bash
# En fvx-backend/ con docker compose en marcha
docker compose exec web python manage.py makemessages -l en -l es
docker compose exec web python manage.py compilemessages
```

Detalle: [`../../fvx-backend/docs/i18n.md`](../../fvx-backend/docs/i18n.md).

## Añadir un idioma

1. Añadir el código en `availableLangs` y en el tipo `FvxUiLang` en `src/app/core/i18n/locale-storage.ts`.
2. Crear `public/assets/i18n/<codigo>.json` (puedes copiar `en.json` como plantilla).
3. Añadir opción al selector de idioma en `config-user.component.ts` (método `langSegmentItems()`), dentro de `app-config-user`.
4. Backend: añadir `('xx', '…')` a `LANGUAGES` en `core/settings.py` y, si añades cadenas traducibles en Python, ejecutar `makemessages` / `compilemessages` **con Docker** (ver resumen al final de este documento).

## Fechas y locale

El formateo de fechas tiene **dos motores** y conviene saber cuál usa cada cosa (`app.config.ts`):

- **`LOCALE_ID` (Angular `DatePipe`, `| date`)** — se fija al idioma persistido en el **bootstrap** (`registerLocaleData(localeEs)` + `provideValue(LOCALE_ID, …)`). Es **estático**: el `DatePipe` toma el idioma con el que cargó la app y **no reacciona** al toggle de idioma en runtime (limitación de Angular). Antes no se proveía → todo `| date` salía en `en-US` aunque la app estuviera en español.
- **`DateAdapter` (date-fns, Material)** — sincronizado con el idioma en **runtime** vía `transloco.langChanges$` en el appInitializer. Los componentes de fecha que deben reaccionar al toggle SIN recargar (calendarios, footer de `app-calendar`/`app-date-picker`) formatean con `dateAdapter.format(...)`, **no** con el `DatePipe`.

**Regla práctica:** si necesitas que un texto de fecha cambie de idioma al togglear sin recargar, usa el `DateAdapter` (o un componente que ya lo haga). Para fechas que solo deben respetar el idioma de carga, `| date` basta.

Detalles relacionados:
- `FVX_DATE_FORMATS` define `dateInput: 'dd-MM-yyyy'` y, para `mat-timepicker` (Material 21), **`timeInput`/`timeOptionLabel: 'HH:mm'` son obligatorios** (sin ellos el timepicker lanza error y el campo de hora no renderiza).
- Tokens M3 `--mat-sys-primary`/`--mat-sys-tertiary` se sobreescriben a `--fvx-link` global en `styles.scss` (el tema base de Material es azure/cyan).
- Claves i18n añadidas: `datePicker.time`, `layout.goHome`.

## Resumen: qué ejecutar con Docker

**Solo cadenas Angular (JSON):** no exige comandos Docker; edita `public/assets/i18n/en.json` y `es.json` y recarga `ng serve`.

**Catálogos Django (tras tocar `gettext_lazy` / `_()` en Python):** en **`fvx-backend/`**:

```bash
docker compose exec web python manage.py makemessages -l en -l es
docker compose exec web python manage.py compilemessages
```

**Comprobar build del front en contenedor** (desde **`fvx-frontend/`**, servicio `web`):

```bash
docker compose exec web npm run build
```

Levantar entornos: [`../README.md`](../README.md) del front y [`../../fvx-backend/README.md`](../../fvx-backend/README.md) del backend (red `fvx_shared`, puertos, etc.).
